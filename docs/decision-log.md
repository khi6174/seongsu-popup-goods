# 의사결정 로그 (Decision Log)

> 프로젝트에서 내린 **중요한 결정**을 시간순으로 append 합니다.
> 각 항목은 아래 형식을 따릅니다. 이 로그는 나중에 **MCP 지식베이스의 핵심 소스**가
> 됩니다 — "왜 그렇게 했는가"를 기계와 사람이 함께 되짚을 수 있도록 구조를 유지하세요.
>
> **형식**
> ```
> ## [번호] 제목  (YYYY-MM-DD)
> - 상황(Context): 어떤 문제/질문 앞에 있었나
> - 선택지(Options): 고려한 대안들
> - 결정(Decision): 무엇을 골랐나
> - 근거(Why): 왜 그걸 골랐나
> - 영향(Consequences): 이 결정으로 뒤따르는 것 / 미룬 것
> ```

---

## [0001] 프로젝트 기록·메모리 체계 수립  (2026-07-02)
- **상황(Context)**: 쇼핑몰 구축을 시작하며, 학습 과정과 의사결정을 나중에 되짚고
  MCP로 재활용할 수 있도록 기록 체계가 필요했다.
- **선택지(Options)**:
  1. 하나의 거대한 로그 파일에 모두 기록
  2. 목적별로 분리된 MD 문서 + Claude 프로젝트 메모리 병행
- **결정(Decision)**: 2번 — `docs/`에 목적별 MD(overview / decision-log /
  learning-log / planning) + Claude 메모리(배경·사용자·작업규칙)를 병행.
- **근거(Why)**: 원자적·태그 가능한 문서가 (1) 사람이 읽기 좋고 (2) 추후 MCP가
  파싱·서빙하기 좋다. 메모리는 세션이 바뀌어도 배경이 유지되게 한다.
- **영향(Consequences)**: 앞으로 중요한 결정이 나올 때마다 이 로그에 항목을 추가한다.
  다음 단계는 기획(요구사항 정의)부터 시작한다.

## [0002] 구축 방식과 학습 초점 결정  (2026-07-02)
- **상황(Context)**: 기획 전에 프로젝트의 큰 방향(어떻게 만들지, 무엇을 중점적으로
  배울지)을 정해야 이후 논의가 정확해진다.
- **선택지(Options)**: 구축 방식 = {직접 코드 / 노코드 플랫폼 / 하이브리드}.
- **결정(Decision)**:
  - 구축 방식 = **직접 코드로 개발**.
  - 학습 초점 = **프론트엔드 + 백엔드/데이터 + AI 기능 접목 + 전체 흐름 경험**(전 과정).
  - 판매 품목 = **미정** (예시를 보며 결정 예정).
- **근거(Why)**: 학습이 진짜 목적이므로, 전 과정을 직접 만들어보는 것이 가장 남는 게
  많다. AI 기능을 처음부터 설계에 포함시키기로 한다.
- **영향(Consequences)**: 기술 스택은 학습에 좋고 AI 연동이 쉬운 것으로 고른다(디자인
  단계에서 확정). 다음 할 일: **제품 컨셉(무엇을 파는가)** 결정.

## [0003] 제품 컨셉 = 취향 추천 향수 몰  (2026-07-02)
- **상황(Context)**: 학습 프로젝트에 어떤 AI 기능을 자연스럽게 담을지가 품목 선택으로
  결정된다. 세 가지 예시(A: 취향추천 향수몰 / B: 디지털 다운로드몰 / C: AI 챗봇 편집숍)를
  놓고 비교했다.
- **결정(Decision)**: **A — 취향 추천 향수/화장품 몰**.
- **근거(Why)**: "문답 → AI 추천"이 몰의 핵심 가치가 되어 AI 학습 목표와 잘 맞고,
  카탈로그가 작아 초심자가 MVP를 완성하기 좋다.
- **영향(Consequences)**: 첫 기획 산출물 `docs/planning/01-product-concept.md` 초안 작성.
  다음: MVP 범위(회원/결제 포함 여부)와 요구사항 확정.

## [0004] MVP 범위 = 비회원 주문 + 모의 결제  (2026-07-02)
- **상황(Context)**: MVP를 가볍게 유지하려면 '핵심(추천+커머스 흐름) 먼저, 복잡한 건
  나중' 원칙에서 회원/결제 범위를 정해야 했다.
- **결정(Decision)**:
  - 회원가입/로그인 = **MVP 제외**(비회원 주문). 인증은 2차 학습.
  - 결제 = **모의(mock)**, '주문 완료'까지만. 실제 PG 연동은 2차.
- **근거(Why)**: 인증과 실제 결제(PG)는 난이도·행정 절차가 커서 학습 흐름을 막는다.
  먼저 상품→추천→장바구니→주문의 전체 골격을 완성해 성취를 얻는 것이 우선.
- **영향(Consequences)**: 데이터 모델에서 회원 테이블은 뒤로 미룬다. 다음 산출물:
  요구사항 상세화 + 화면 목록(정보구조), 이어서 데이터 모델 초안.

## [0005] 컨셉 전환 = 성수동 팝업 굿즈 글로벌 몰  (2026-07-02)
- **상황(Context)**: 사용자가 컨셉을 재검토, 향수 몰에서 방향을 바꾸기로 했다.
- **결정(Decision)**: 새 컨셉 = **성수동 팝업스토어 방문객을 겨냥한 굿즈 글로벌
  쇼핑몰**. 기존 향수 몰 컨셉(`01-product-concept.md`)은 보류(superseded).
- **근거(Why)**: 성수동은 팝업이 활발하고 국내외 방문객이 많아, "팝업 굿즈 + 글로벌
  (외국인 관광객)"이라는 뾰족한 타겟과 차별점을 만들 수 있다.
- **영향(Consequences)**: 기획 발굴을 위해 **서브 에이전트를 띄워 십문십답**
  (`00-discovery-10qa.md`)을 생성. 사용자가 답하면 새 제품 컨셉 정의서를 작성하고
  화면 목록·데이터 모델로 진행한다. 앞선 결정 0004(비회원+모의결제)의 MVP 경량화
  원칙은 새 컨셉에도 유지.

## [0006] 십문십답 추천안 전체 채택 → 컨셉/화면 확정  (2026-07-02)
- **상황(Context)**: 서브 에이전트가 만든 십문십답에 대해, 초심자로서 각 항목의 전문가
  추천 답안을 채택할지 결정해야 했다.
- **결정(Decision)**: **10개 추천 답안을 전체 채택**. 이를 바탕으로:
  - 제품 컨셉 정의서 `01-product-concept.md` v1.0 확정.
  - MVP 화면 목록/흐름 `02-screens-and-flow.md` v1.0 작성(홈·팝업상세·상품상세·
    장바구니·체크아웃·주문완료 + 전역 AI챗봇·언어전환).
- **근거(Why)**: 추천안이 컨셉(성수동·글로벌·팝업굿즈)과 타겟(외국인 관광객)에 일관되게
  정렬돼 있고, 초심자가 빠르게 뼈대를 잡아 다음 단계로 넘어가는 것이 학습에 유리.
- **영향(Consequences)**: 다음 산출물 = **데이터 모델 초안**(팝업·상품·주문·장바구니).
  그 뒤 기술 스택 결정 → 개발 단계 진입.

## [0007] 데이터 모델 & 정보구조(IA) 확정  (2026-07-02)
- **상황(Context)**: 화면 목록을 실제 저장 구조와 탐색 구조로 구체화할 차례.
- **결정(Decision)**: `03-data-model.md`, `04-information-architecture.md` v1.0 작성.
  핵심 설계 판단:
  - 엔티티 = Popup / Product / CartItem / Order / OrderItem (+ 배송비·환율·i18n은 설정).
  - **장바구니는 비회원이라 브라우저 세션/localStorage에 귀속**(User 테이블 없음).
  - **주문은 스냅샷 원칙** — OrderItem에 주문 당시 상품명·가격을 복사 저장.
  - **팝업 상태(열림/마감)는 저장이 아니라 날짜로 계산**.
  - 다국어는 MVP에서 `_ko`/`_en` 두 필드로 단순화(과설계 금지).
  - IA 조직 축 = **1차 팝업별 · 2차 카테고리별 · 상태(열림/마감)**, 탐색 경로 3갈래
    (둘러보기·카테고리·챗봇)가 상품 상세로 수렴. 언어 전환 상시 노출.
- **근거(Why)**: 비회원·모의결제 MVP에 맞춰 최소 엔티티로 전체 커머스 흐름을 학습.
  스냅샷·계산상태는 실무에서 중요한 데이터 설계 습관이라 처음부터 몸에 익힌다.
- **영향(Consequences)**: 기획~정보설계 완료. 다음 = **기술 스택 결정** → 개발 진입.

## [0008] 사용자 시나리오(Full) 작성  (2026-07-02)
- **상황(Context)**: 기획을 실제 사용 이야기로 검증하고 개발 대본을 만들 차례.
- **결정(Decision)**: `05-user-scenarios.md` v1.0 작성. 페르소나 3명(외국인 관광객
  Emma·국내 트렌드세터 지원·원격 구매자 Kenji)과 시나리오 A(해피패스)·B(품절 대안
  경로)·C(원격 배송)·부분 시나리오·예외 체크리스트를 각 단계마다 화면 ID·데이터에 연결.
- **근거(Why)**: 시나리오를 화면·데이터에 매핑하면 "이 화면에서 무엇을 만들어야 하는지"가
  명확해지고, 예외 상황(품절·배송불가)을 미리 잡아 개발 누락을 막는다.
- **영향(Consequences)**: 기획 단계 산출물(00~05) 완결. 다음 = **기술 스택 결정**.

## [0009] 기술 스택 확정 & 전담 서브 에이전트 배치  (2026-07-02)
- **상황(Context)**: 개발 진입 전, 스택을 정하고 도구·기술별 전담 서브 에이전트를 배치해
  이후 개발을 역할별로 위임할 체계가 필요했다.
- **결정(Decision)**:
  - 스택 = **Next.js(App Router)+TypeScript / Tailwind / next-intl / Prisma+SQLite /
    Anthropic Claude API / Vercel(이후)**. 상세는 `06-tech-stack.md`.
  - `.claude/agents/`에 전담 에이전트 5개 배치: **frontend-engineer, backend-engineer,
    database-engineer, ai-engineer, devops-engineer**. 각 에이전트는 담당 스택과
    관련 기획 문서(01~06)를 참조하고, 초심자 코칭 방식으로 작동하도록 시스템 프롬프트 작성.
- **근거(Why)**: 하나의 Next.js 생태계에서 풀스택+AI를 학습하기 좋고 자료가 풍부.
  역할별 에이전트로 관심사를 분리하면 초심자가 "누구에게 무엇을 맡길지" 명확해진다.
- **영향(Consequences)**: 다음 = **devops-engineer로 프로젝트 초기 세팅** →
  database-engineer로 스키마/시드 → 화면·API·AI 순으로 개발.

## [0010] 개발 파이프라인 1~4단계 구현 (동작하는 MVP)  (2026-07-02)
- **상황(Context)**: 기획·설계·스택 확정 후, 실제로 동작하는 MVP를 만들 차례.
  디자인은 나중에 무드보드로 입히기로 하고, 기능 우선의 중립 스타일로 구현.
- **결정/구현(Decision)**:
  - **1단계 세팅**: Next.js 16(App Router, TS, Tailwind v4) 스캐폴드를 임시폴더에
    만들어 프로젝트 루트로 병합(기존 docs·.claude 보존). Prisma 7·Anthropic SDK 등 설치.
  - **2단계 DB**: Prisma 스키마(Popup/Product/Order/OrderItem) + 마이그레이션 + 시드
    (팝업 3, 굿즈 7, 품절 1). Prisma 7은 **드라이버 어댑터 필수** →
    `@prisma/adapter-better-sqlite3` 사용.
  - **3단계 화면/백엔드**: 홈·팝업상세·상품상세·장바구니·체크아웃·주문완료 +
    카테고리 탐색(/shop). 주문 API(`/api/orders`)는 **DB에서 가격 재계산·재고 확인·
    스냅샷 저장·배송비 계산**. 장바구니는 localStorage(비회원).
  - **4단계 AI**: `/api/chat`(다국어 챗봇, 카탈로그 컨텍스트 주입) +
    `/api/translate`(한→영). API 키 없으면 친절한 폴백 메시지.
  - **검증**: `npm run build` 타입체크 통과. 서버 기동 후 전 라우트 200,
    주문 생성(200, SS-20260702-0001)·품절 거부(409)·챗봇 폴백 정상 확인.
- **기술 판단(중요)**:
  - **i18n는 next-intl 라우팅 대신 쿠키 기반 경량 방식** 채택(초심자 가독성·MVP 속도).
    확장 시 next-intl 라우팅으로 승격 가능. (설치는 되어 있으나 미사용)
  - **Google 폰트(next/font) 제거** → 네트워크 의존 없이 시스템 폰트 사용.
  - 결제는 실제 없음(모의). 이미지 없어 카테고리 이모지+그라데이션 플레이스홀더.
- **근거(Why)**: 기능이 도는 골격(walking skeleton)을 먼저 완성해 프론트·백·DB·AI를
  한 바퀴 경험. 디자인·실결제·회원은 이후 단계.
- **영향(Consequences)**: 다음 = **디자인 단계**(무드보드 URL → UI 입히기) 또는
  실제 ANTHROPIC_API_KEY 넣고 챗봇 실동작 확인. 실행법은 README 참고.

## [0011] 디자인 무드보드 'Playful Pop' 확정  (2026-07-02)
- **상황(Context)**: 디자인 단계 진입. 사장님(초심자)이 방향 선택에 자신이 없어,
  레퍼런스(Dribbble/Pinterest)로 방향 3안 추천 → 자리 비움 → 추천안(🅐 Playful Pop)으로
  진행. 요청대로 **디자인 전문가 서브 에이전트(design-lead)** 배치 + **페르소나 3명
  (Emma·지원·Kenji) 의견을 각각 수집**해 종합.
- **결정(Decision)**: 디자인 방향 = **Playful Pop**. 중심 원칙 = **"귀여움(Playful) ↔
  신뢰(Trust)를 영역별로 분리"** — 둘러보기/카드/캐릭터는 파스텔+포인트색+마이크로
  인터랙션, **돈·배송·결제 영역은 흰 카드+진한 잉크 고대비 + "No hidden fees"**.
  산출물 = `docs/design/moodboard.html`(9개 섹션, WCAG AA 대비 검증).

## [0013] 배포/DB/레포 CLI 도구 설치  (2026-07-02)
- **상황(Context)**: 배포(Vercel)·DB(Supabase)·레포 관리(GitHub) 준비를 위해 CLI 3종 설치 요청.
- **결정(Decision)**:
  - **Vercel CLI** — `npm i -g vercel` 전역 설치(v54.18.7). 표준 방식.
  - **GitHub CLI(gh)** — 이미 설치돼 있었음(v2.89.0). 별도 조치 없음.
  - **Supabase CLI** — **프로젝트 dev 의존성**으로 설치(`npm i -D supabase`, v2.109.0) →
    `npx supabase ...` 로 실행. package.json에 기록됨.
- **근거(Why)**: Supabase CLI는 **Windows에서 npm 전역 설치를 공식 미지원**이고 winget에도
  없음. 남은 선택지는 Scoop(전역) vs dev 의존성. 공개·학습 프로젝트라 **clone만 하면 동일 버전이
  재현되는 dev 의존성**을 선택(관리자 권한 불필요, 공식 지원 경로). 대신 명령은 `npx supabase`.
- **영향(Consequences)**: 배포·DB 마이그레이션 도구 준비 완료. 아직 **DB를 Supabase(Postgres)로
  전환한 것은 아님** — 현재는 여전히 SQLite. 다음 후보 = GitHub 레포 생성(gh) → Vercel 연결 →
  (필요 시) Prisma 데이터소스를 Supabase Postgres로 전환.

## [0015] DB 전환: SQLite → Supabase(PostgreSQL)  (2026-07-02)
- **상황(Context)**: Vercel 서버리스 배포 전, 파일 기반 SQLite는 서버리스에서 못 쓰므로
  클라우드 Postgres로 이전 필요. 배포 순서 1(커밋)→2(DB전환)→3(배포) 중 2단계.
- **결정(Decision)**:
  - Supabase 프로젝트를 **대시보드에서 생성**, 리전 **Seoul(ap-northeast-2)**. (계정·비밀번호는
    사장님이 직접 — 비밀번호는 특수문자 이슈로 영문+숫자로 재설정 후 `.env`에 로컬 입력)
  - Prisma 데이터소스 `sqlite`→`postgresql`. **Prisma 7.8부터 접속 URL은 스키마가 아니라
    `prisma.config.ts`의 `datasource.url`**에 둠 → 마이그레이션용 = `DIRECT_URL`(세션 풀러 5432).
  - 런타임 접속은 드라이버 어댑터 **`@prisma/adapter-pg`(PrismaPg)** + `DATABASE_URL`
    (**트랜잭션 풀러 6543, `pgbouncer=true`** — 서버리스 적합). `better-sqlite3` 어댑터 제거.
  - 스키마 히스토리 대신 **`prisma db push`**로 테이블 생성(Supabase 풀러에서 shadow DB 이슈 회피,
    MVP엔 충분). seed는 `DIRECT_URL`로 접속.
- **근거(Why)**: 서버리스=트랜잭션 풀러, 마이그레이션=세션 풀러가 Supabase+Prisma 표준. 드라이버
  어댑터는 Prisma 7 필수. db push는 초심자·프로토타이핑에 단순하고 shadow DB 권한 문제 없음.
- **검증(Verify)**: db push·seed 성공(팝업3·굿즈7). 앱 런타임에서 홈/‧shop이 Supabase 데이터
  렌더(읽기 OK), 주문 생성 API 200·SS-20260702-0001(쓰기·트랜잭션 OK, 풀러 경유). `npm run build` 통과.
- **영향(Consequences)**: 로컬·배포 모두 Supabase Postgres 사용. `.env`는 커밋 안 됨(비밀번호 로컬).
  다음 = 3단계 **Vercel 배포**: 레포 연결 + 환경변수(DATABASE_URL·DIRECT_URL·ANTHROPIC_API_KEY) 등록.
